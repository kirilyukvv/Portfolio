# Ideas

Case study template → [project-page-structure.md](project-page-structure.md)

## Team avatars on projects

- Show that work is not solo — on **each project** (home grid cards and case study pages).
- **Pixel-art avatars** for everyone who contributed, including me.
- Overlapping avatar stack + **headcount** (e.g. “5 people”) so it’s clear how many worked on the project.
- Optional later: names/roles on hover or always visible; replace placeholder pixel art with real assets per person.

## Manager experience page

- Dedicated page (e.g. `management.html` or `lead.html`) — not buried in CV; linked from home hero or nav alongside CV.
- Voice: **lead / people manager**, complementary to project case studies (craft) and CV (timeline).
- **Skills** — what I actually do as a manager: hiring, 1:1s, feedback, roadmap alignment, design ops, cross-functional leadership, mentoring, stakeholder management. Short labels + one line each, not a buzzword cloud.
- **Achievements** — concrete wins with context: team built or grown, process introduced, quality bar raised, conflict resolved, designer promoted, delivery under constraint. Prefer “what changed for the team / product” over title-only claims.
- **Fails** — honest misses: what went wrong, what I learned, what I’d do differently. Keeps the page credible; pairs with achievements (no humblebrag-only section).
- Structure idea: snapshot at top (team size, tenure as lead, domains) → Skills → Achievements (bullets or short stories) → Fails → optional **Principles** (how I manage, 3–5 lines).
- Same visual system as the rest of the site (mono, geeky symbols, no emoji). Optional: tie to **team roster** / avatars when talking about people I’ve led.
- Privacy: first names or roles only if needed; no client secrets in fail stories.

## Geeky symbols instead of emoji

- Use **typographic / Unicode symbols** for accents and UI chrome — not emoji (feels casual; clashes with mono + terminal vibe).
- Fits **JetBrains Mono** and the existing language: middle dots `·`, arrows `→`, maybe `▸` `◆` `※` `⌁` `λ` `∴` for labels, bullets, CTAs, section markers.
- Where emoji might have been used (highlights, tags, “new”, team, links), pick a **small fixed set** and reuse everywhere so it reads as a system, not decoration.
- Optional: one **symbol font** or ligature set only if we need icons emoji would cover (e.g. mail, external link) — still line-based, not colorful pictographs.
- Watch **accessibility**: keep real text in `aria-label` / visible copy; symbols are visual shorthand only.
